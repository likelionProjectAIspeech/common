# 협업 전략 및 개발 방향

## 프로젝트 개요

AI Speech는 사용자가 PPT, PDF, DOCX 형식의 발표 자료를 업로드하면 AI가 자료를 분석하여 슬라이드별 발표 시간을 배분하고 발표 대본을 생성하는 서비스입니다.

사용자는 스마트폰을 이용해 실제 발표처럼 연습할 수 있으며, 시스템은 음성, 자세, 시선, 반복 행동을 분석합니다. 이후 AI 에이전트가 발표를 평가하고 예상 질문과 답변 개선 방향을 제공합니다.

## 전체 서비스 흐름

```text
발표 자료 업로드
-> 발표 자료 및 슬라이드 분석
-> 슬라이드별 발표 시간 배분
-> 발표 대본 생성
-> 스마트폰 발표 리허설
-> 음성 및 영상 분석
-> AI 에이전트 평가
-> 예상 질문 및 답변 분석
-> 종합 리포트 제공
-> 대본 수정 및 재연습
```

## 기술 스택

### Frontend

- React Native
- 스마트폰 카메라 및 마이크 연동
- 발표 자료 및 대본 조회
- 발표 녹음 및 영상 촬영
- 분석 결과 및 종합 리포트 시각화
- 예상 질문 확인 및 답변 녹음
- 사용자별 발표 기록과 개선 정도 비교

### Backend

- Python
- FastAPI
- RQ
- Redis
- PostgreSQL

FastAPI는 프론트엔드 요청을 처리하는 REST API 서버로 사용합니다. RQ와 Redis는 오래 걸리는 AI 분석 작업을 비동기적으로 처리하는 데 사용합니다.

주요 비동기 처리 대상은 다음과 같습니다.

- 발표 자료 분석
- 음성 파일 변환 및 분석
- 영상 프레임 분석
- 자세 및 시선 분석
- AI 에이전트 평가
- 예상 질문 생성
- 종합 리포트 생성

### AI 및 영상 분석

| 기술 | 사용 목적 |
| --- | --- |
| OpenAI Whisper API | 발표 음성을 텍스트로 변환 |
| OpenAI GPT API | 대본 생성, 평가, 예상 질문 생성, 답변 분석 |
| YOLO Pose | 발표자의 신체 관절과 자세 추적 |
| OpenCV | 영상 프레임 처리 및 행동 분석 |
| MediaPipe | 얼굴, 시선, 머리 방향, 손동작 분석 |
| FFmpeg | 영상 및 음성 분리, 압축, 형식 변환 |

영상 분석에서는 단순히 특정 행동의 발생 여부만 확인하지 않고, 반복 횟수, 지속 시간, 발생 구간, 발표 내용과의 연관성을 함께 분석합니다.

### 문서 분석

| 파일 형식 | 처리 기술 |
| --- | --- |
| PPTX | python-pptx |
| DOCX | python-docx |
| PDF | PyMuPDF |
| PPT, DOC | LibreOffice Headless 변환 후 처리 |

문서에서 추출하는 주요 정보는 다음과 같습니다.

- 슬라이드 제목
- 본문 텍스트
- 표와 그래프
- 이미지
- 발표 자료의 전체 구조
- 슬라이드 간 연결성
- 중복 내용
- 핵심 키워드
- 설명이 부족한 부분

## 시스템 아키텍처

```text
React Native App
  |

  |- GPT API
  |- YOLO Pose
  |- OpenCV
  |- MediaPipe
  |- FFmpeg
  |
PostgreSQL
```

## 데이터 처리 원칙

- API Key를 React Native 앱에 하드코딩하지 않습니다.
- OpenAI API 호출은 반드시 FastAPI 서버를 통해 처리합니다.
- 발표 자료, 영상, 음성 파일은 접근 권한을 확인한 뒤 제공합니다.
- 사용자별 데이터가 분리되도록 모든 발표 프로젝트에 사용자 식별자를 연결합니다.
- 영상과 음성 분석은 비동기 작업으로 처리합니다.
- 분석 중인 작업은 상태값으로 관리합니다.

작업 상태 예시는 다음과 같습니다.

- `PENDING`
- `PROCESSING`
- `COMPLETED`
- `FAILED`

## 개인정보 및 파일 보안

- 사용자 동의 없이 음성과 영상을 수집하지 않습니다.
- 발표 자료, 영상, 음성의 수집 목적을 명확하게 고지합니다.
- 사용자가 파일 삭제를 요청할 수 있도록 기능을 제공합니다.
- 사용자별 파일 접근 권한을 분리합니다.
- 파일 다운로드 URL은 영구 공개 URL이 아닌 제한된 URL 사용을 검토합니다.
- 분석 완료 후 원본 영상과 음성을 자동 삭제하는 옵션을 검토합니다.
- 데이터 보유 기간을 개인정보처리방침에 명시합니다.

## GitHub 협업 규칙

### Pull Request

- 모든 기능 개발은 별도 브랜치에서 진행합니다.
- 작업 완료 후 `dev` 브랜치를 대상으로 Pull Request를 생성합니다.
- 코드 리뷰 후 병합합니다.
- `main` 브랜치에 직접 push하지 않습니다.
- 배포 가능한 상태로 검증된 코드만 `dev`에서 `main`으로 병합합니다.

### 브랜치 전략

| 브랜치 | 용도 |
| --- | --- |
| `feat/...` | 새로운 기능 개발 |
| `fix/...` | 버그 및 오류 수정 |
| `dev` | 개발 기능 통합 및 테스트 |
| `main` | 최종 배포 버전 |

### 작업 흐름

```text
dev 브랜치에서 기능 브랜치 생성
-> 기능 개발
-> 기능 브랜치 push
-> dev 대상 Pull Request 생성
-> 코드 리뷰
-> dev 병합
-> 통합 테스트
-> main 병합 및 배포
```
